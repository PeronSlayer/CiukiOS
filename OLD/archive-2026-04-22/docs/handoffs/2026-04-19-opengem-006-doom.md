# 2026-04-19 — OPENGEM-006: DOOM Path Readiness

## Context and goal
Phase 6 of the OpenGEM UX roadmap
([docs/roadmap-opengem-ux.md](../roadmap-opengem-ux.md)) — final
phase. Wire the catalog, OpenGEM session bridge, and shell dispatch
so that a user-supplied shareware DOOM binary staged on the FAT
image is (a) discovered by the Phase 4 app catalog, (b) launched
via OpenGEM with a clear telemetry boundary, and (c) validated by a
fixture-gated end-to-end harness that SKIPs cleanly when fixtures
are absent (the normal CI case under CiukiOS licensing policy).

No version bump (`CiukiOS Alpha v0.8.7`). No changes to `main`.
DOOM binaries/WADs remain **user-supplied**.

## Decisions made
- **Catalog probe lives in stage2.c, not shell.** Running the probe
  immediately after `app_catalog_init(handoff)` guarantees the
  DOOM-discovery markers appear in the boot log regardless of
  which path the user takes (desktop, dock, `opengem`, direct
  shell). One authoritative point of truth.
- **Launch marker lives in `shell_run_from_fat()`.** The runtime
  EXE/COM dispatch for interactive OpenGEM sessions passes through
  this helper. A case-insensitive match on the target basename
  (`DOOM.EXE`) is the simplest, stable correlation point.
- **Two-tier harness: static always + runtime gated.** The static
  invariants (7 assertions checking that the markers exist in the
  source and that the probe is wired) run on every `make
  test-doom-via-opengem` invocation. The runtime assertions run
  only when a user explicitly opts in by setting
  `CIUKIOS_DOOM_FIXTURES_DIR`. Default CI path SKIPs cleanly with
  a PASS exit code.
- **`CIUKIOS_DOOM_BOOT_LOG` overridable.** A separate env var lets
  the user point the harness at any captured boot log (from QEMU
  serial redirection, a physical boot, etc.) without forcing a
  hardcoded path.
- **No redistribution.** `docs/boot-to-doom-via-opengem.md` opens
  with a licensing notice. The harness does not embed, copy, or
  reference any DOOM binary path relative to the repository. The
  `CIUKIOS_DOOM_FIXTURES_DIR` is an absolute user path.
- **No new services ABI.** Existing `ciuki_services_t` is frozen.
  All markers emit via `serial_write()`. DOOM as a DOS extender
  client does not need an ABI change; the INT 33h hook from
  Phase 5 is the relevant extension point if a future GEMVDI
  wants to synthesize pointer events on DOOM's behalf.

## Risks and next step
- **DOS/4GW extender bootstrap incomplete.** Real DOOM is a
  DOS/4GW client. The M6 DPMI smoke programs
  (`com/m6_dpmi_*/`) exercise the building blocks; Phase 6's
  harness expects "stage reached: menu" but will report a gap —
  not block — if the extender path cannot attach. Documented in
  the gap list.
- **Mode 13h corner cases.** DOOM's direct VGA writes exercise
  scroll/double-buffer paths that currently have known rough
  edges; tracked in
  `docs/copilot-task-codex-sr-video-003.md`.
- **Audio absent.** Sound Blaster/AdLib emulation is out of scope
  for the OpenGEM UX roadmap. DOOM will fall back to silent mode.
- **Fixture-gated runtime assertions are not exercised in CI.**
  Without fixtures, the runtime DOOM markers cannot be verified
  in automation. Mitigation: the static invariants confirm the
  markers are compiled in, and a user with fixtures can exercise
  the full harness locally.
