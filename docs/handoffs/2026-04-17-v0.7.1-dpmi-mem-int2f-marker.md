Continue roadmap progress toward real DOS DOOM binaries. Extend the DPMI synthetic-service chain by adding an allocate-memory-block (AX=0501h) smoke, extend the BIOS compatibility markers with INT 2Fh (multiplex), and fix two Makefile hygiene regressions inherited from v0.7.0.

## Decisions
- DPMI AX=0501h returns a synthetic 32-bit handle and linear address (upper word BX/SI=0x0010, lower CX/DI=0x0000). Enough for DPMI contract validation; real extenders will still need a real allocator.
- Zero-size request (BX==0 && CX==0) mapped to DPMI error 8021h ("invalid linear address" / generic failure) with CF=1 to exercise the error path.
- `CIUKMEM.EXE` keeps the same slim layout pattern as the LDT smoke (inline zero_regs + goto done) to stay under the 512-byte MZ hard limit.
- Documentation gate list in `documentation.md` was rewritten because the prior list contained a typo (`test-mm6-dpmi-ldt-smoke`) and broken numbering; cleaned to 18 ordered entries.

## Risks
- Synthetic allocator only; a real DOS extender allocating then writing to the returned linear address will still fault. Promotion to a real allocator must coordinate with the M6_PMEM reserved range and add overlap checks.
- Static-fallback validation cannot catch a runtime regression where the synthetic handler silently changes its returned shape; the shell-side code path is exercised only by source grep.

## Next Steps
1. Promote the synthetic AX=0501h handler to a real allocator backed by the M6_PMEM pool with boundary checks and an `[m6] dpmi mem allocate base=... size=...` runtime trace.
2. Add AX=0502h (free memory block) matching symbolic handle semantics.
3. Consider a paired runtime test path that boots far enough to hit the handler and emits the marker into stage2 logs so the grep fallback becomes a true regression net.
